# Contributing to NeuroLearn

NeuroLearn is developed by a three-person team with AI coding agents assisting implementation. This workflow is designed to keep human ownership clear, make agent-generated changes reviewable, and prevent shared contracts from drifting across workstreams.

## Project Ownership

The roadmap is organized into three coherent workstreams:

| Workstream | Primary responsibility |
|---|---|
| Platform & Product Foundation | Identity, authorization, schema, APIs, jobs, privacy, security, testing, observability, and deployment |
| Knowledge & Content AI | Document ingestion, indexing, concept graph, curriculum, grounded lessons, retrieval, tutor, and AI evaluation |
| Adaptive Learning & Learner Experience | Onboarding, diagnostics, assessments, mastery, recommendations, learning sessions, analytics, accessibility, and academic evaluation |

Ownership means responsibility for the contract and final integration. It does not prevent another member from reviewing, pairing, or contributing a scoped change.

## Sources of Truth

Before starting work, read:

1. The Linear issue, parent workstream, milestone, and blockers.
2. `AGENTS.md` for engineering rules.
3. `SYSTEM_ARCHITECTURE.md` for boundaries and invariants.
4. The affected implementation and tests.

The issue defines the approved outcome. Code defines present behavior. Architecture documents define intended boundaries. If they conflict, resolve the conflict visibly instead of silently encoding an assumption.

## Issue Readiness

An implementation issue is ready when it has:

- a clear outcome and owner
- scope and non-goals
- acceptance criteria
- known dependencies and blockers
- affected contracts or components
- enough detail to test completion

Large roadmap issues should be decomposed into child tasks that can normally be completed in three to five focused working days. Equal ticket counts are not used as a proxy for equal effort; estimate after decomposition.

## Branches and Pull Requests

Create one branch per Linear issue:

```text
<linear-id>-<short-description>
```

Examples:

```text
kis-8-secure-auth
kis-14-document-extraction
kis-24-mastery-model
```

Keep a pull request focused on one coherent contract or vertical slice. Avoid mixing refactors, dependency upgrades, formatting sweeps, and product behavior unless they are required by the issue.

Every pull request should include:

- Linear issue link
- problem and implemented outcome
- important design decisions
- API/schema/migration/configuration changes
- security and privacy considerations
- verification commands and results
- screenshots or recordings for meaningful UI changes
- known limitations and follow-up issues

AI-generated code is held to the same standard as human-written code. “Generated by an agent” is neither evidence of correctness nor a reason to skip explanation.

## Contract-First Collaboration

Shared contracts are the main coordination points among the three workstreams:

- identifiers and ownership rules
- entity and state names
- database constraints and migrations
- API request/response schemas
- job names, payloads, and states
- event names and version fields
- source provenance and citation structures
- assessment evidence and mastery inputs

When a change affects another workstream:

1. Propose the contract in the issue or pull request.
2. Tag the affected owner.
3. Add or update contract tests and fixtures.
4. Merge the provider side before removing consumer mocks.
5. Update `SYSTEM_ARCHITECTURE.md` when the boundary or data flow changes.

Avoid coordinating through undocumented payloads in chat alone.

## Working with Blockers

Dependencies are expected in the source-to-learning pipeline. A blocked issue should identify whether it is:

- **Integration-blocked:** isolated work can proceed using an agreed interface, fake, or fixture.
- **Decision-blocked:** a shared design choice must be resolved before implementation.
- **Execution-blocked:** required code or infrastructure truly does not exist yet.

Do not leave a team member idle for an integration blocker. Work on isolated logic, UI states, tests, fixtures, evaluation data, or documentation. Do not invent and merge a shared contract when the owning decision is unresolved.

## Implementation Standards

### Backend

- Keep domain logic independent of FastAPI request objects.
- Use Pydantic schemas at boundaries and SQLAlchemy models for persistence.
- Enforce authentication and ownership server-side.
- Add an Alembic migration for every schema change.
- Keep long-running work out of request handlers.
- Make jobs and externally retried writes idempotent.
- Return structured errors; do not leak exceptions or secrets.

### Frontend

- Follow App Router conventions.
- Prefer typed shared API access over ad hoc `fetch` calls.
- Do not expose service credentials or internal tokens to the browser.
- Implement complete loading, empty, processing, failure, and retry states.
- Keep components accessible and responsive.
- Do not display hard-coded analytics as real learner data.

### AI and data

- Treat uploaded content as untrusted.
- Validate structured model output before use.
- Preserve source, prompt, model, schema, and validator versions.
- Use deterministic fakes in automated tests.
- Report uncertainty and insufficient evidence.
- Do not copy private learner/source data into issues, prompts, logs, or fixtures.

## Testing and Review

Tests should be added at the lowest useful layer:

- unit tests for deterministic domain rules
- repository/service tests for persistence and ownership
- API tests for authentication, validation, idempotency, and errors
- contract tests for cross-workstream payloads
- component tests for interactive UI states
- end-to-end tests for critical learner journeys
- evaluation fixtures for retrieval, generation, grading, and adaptation

Reviewers should check:

- Does the change satisfy the issue without expanding scope?
- Is present behavior accurately distinguished from target scaffolding?
- Are ownership, privacy, and failure cases covered?
- Is historical/versioned evidence preserved?
- Are migrations safe and contracts backward-compatible where needed?
- Can the behavior be verified without relying on the author’s machine or paid AI calls?
- Were architecture and agent instructions updated if assumptions changed?

At least one teammate should review every product change. Shared schema, authentication, job, event, or AI contract changes should be reviewed by the affected workstream owner.

## Commits

Use concise imperative commit subjects. Include the Linear identifier when practical:

```text
KIS-24 add versioned concept mastery updates
```

Do not commit:

- `.env` files or credentials
- user-uploaded documents or production data
- model-provider tokens
- generated dependency/build directories
- unrelated editor or operating-system files
- database dumps containing personal data

## Documentation Requirements

Update documentation in the same pull request when changing:

- a system boundary or data flow
- an API or event contract
- a persistent entity or state machine
- authentication, authorization, privacy, or retention behavior
- a background job or failure-recovery mechanism
- an implemented-versus-planned architecture status

`README.md` remains the concise product and architecture map. Detailed architecture belongs in `SYSTEM_ARCHITECTURE.md`. Agent-specific operating rules belong in `AGENTS.md`.
